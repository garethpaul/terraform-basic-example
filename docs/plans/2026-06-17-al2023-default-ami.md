# Region-Local Amazon Linux 2023 Default AMI

## Status: In Progress

## Context

The example defaults to a hard-coded Ubuntu 18.04 AMI for `us-east-2` and tells
callers to replace it in every other region. That image generation is obsolete,
the identifier can disappear, and the accompanying user-data script assumes a
BusyBox executable that is not part of the maintained application contract.

AWS publishes region-local Amazon Linux 2023 AMI IDs through Systems Manager
public parameters. The AWS provider can read those parameters without storing
credentials, and Amazon Linux 2023 provides `/usr/bin/python3` as its stable
system Python entry point.

## Priority

1. Replace the obsolete, region-specific default image with the current
   region-local Amazon Linux 2023 x86_64 AMI published by AWS.
2. Preserve explicit AMI overrides and avoid querying the public parameter when
   an override is supplied.
3. Keep the credential-free mocked test boundary and never run a real plan or
   apply from the agent environment.

## Requirements

- Make `ami_id` an optional nullable override while retaining existing 8- and
  17-hex-character validation for explicit values.
- Read the exact AWS public parameter
  `/aws/service/ami-amazon-linux-latest/al2023-ami-kernel-default-x86_64` only
  when `ami_id` is null.
- Use the public parameter's non-sensitive value for `aws_instance.example.ami`;
  do not introduce a secret, writable parameter, or stateful resource.
- Preserve explicit override precedence and prove it with mocked Terraform
  tests that never contact AWS.
- Serve the sample page from a dedicated directory with AL2023's
  `/usr/bin/python3 -m http.server`, retaining `server_port` interpolation and
  `user_data_replace_on_change`.
- Add fail-closed source contracts, mocked data defaults, documentation, and
  completed verification evidence.

## Implementation Units

### U1. Resolve a maintained regional image

- **Files:** `main.tf`, `variables.tf`
- **Outcome:** The default path reads the AWS-owned AL2023 public parameter;
  explicit AMI IDs bypass that data source and remain structurally validated.

### U2. Keep the example startup compatible

- **Files:** `main.tf`
- **Outcome:** User data writes `/var/www/html/index.html` and launches the
  AL2023 system Python HTTP server from that directory.

### U3. Prove both selection paths

- **Files:** `tests/ami_id.tftest.hcl`, `scripts/check-terraform-source.py`
- **Outcome:** Mocked provider data proves the regional default and explicit
  override paths without credentials or infrastructure creation; source
  contracts reject stale AMIs, unconditional lookups, and BusyBox regressions.

### U4. Record the maintained default

- **Files:** `README.md`, `SECURITY.md`, `VISION.md`, `CHANGES.md`, `AGENTS.md`,
  `docs/plans/2026-06-17-al2023-default-ami.md`
- **Outcome:** Operator guidance distinguishes the maintained regional default
  from explicit architecture- or workload-specific overrides.

## Verification

- Focused hygiene and configuration contracts.
- Mocked Terraform tests for the default public parameter and explicit override.
- Repository and external-directory `make check` with read-only provider lock.
- Hostile mutations for nullable validation, conditional lookup, public
  parameter path/value, override precedence, AL2023 startup, mocked data,
  documentation, and completed plan evidence.
- Exact-diff, Terraform generated-artifact, credential-pattern, lock digest,
  staged-path, and whitespace audits.
- One bounded exact-head hosted Terraform snapshot after push; no credentials,
  real AWS plan, or infrastructure apply.

## Scope Boundaries

- Do not change provider versions, instance type, network exposure, tags,
  metadata controls, root-volume encryption, outputs, or workflow permissions.
- Do not resolve the public parameter from the local agent environment.
- Do not claim a real EC2 launch; native coverage remains the mocked Terraform
  plan/apply test boundary.
